Project Description

This is a device manager for SiLA2 laboratory devices. SiLA2 servers that are registered in the network (python-zeroconf) can be automatically detected and added to the manager. The interactive feature browser displays all implemented features and functions of the device and allows the execution of commands from the browser.

This IoT application for SiLA devices is aimed at everybody who is working in a modern laboratory and is interested in lab automation. This is for you, if you're new to SiLA2 and want to play around with your HelloSiLA_server or if your're more advanced and plan on connecting your existing devices and establishing a simple workflow using the device manager scripting environment. Devices overview

The Devices Overview shows all devices that have been added to the device manager. New devices can be added either by using the auto-discovery feature or by manually supplying the connection details. By clicking on the info -icon of one of the devices, a more detailed view pops up. In this view you can browse the devices features and discover the implemented functions of that SiLA device. You can even execute commands and interact with your device directly from there without any additional programming. The list view of the items displays the most important attributes of the devicesw servers such as the connection detail, availability and whether the server is connected or not.

Schedule

The Schedule view shows the calendar that is served by the backend scheduler. The scheduler manages bookings of devices in the backend and stores that information in a database. Reservations can be made manually by a user or indirectly by an experiment (user script). A booking will block a device from being accessed from other sources or users through the device manager. When a device is marked as busy, only the user that booked the device or started the script that made the reservation, will be allowed to send commands to that device through the device detail view. The same holds true for data management. Database settings of devices that are booked are restricted.

Experiments

In the Experiments view, the user can start adn schedule experiments based on provided user scripts (Python 3.6+). Dispatched experiments will reserve the devices for the respective time period and prevent other users from accessing them for that duration. On top of that, a user can make manual reservations. Clicking on an existing triggers a pop-up that displays further information.

My Scripts

The user can upload python scripts for later use as experiments in the My Scripts section. Although the execution of a script is triggered from the "Experiments view", scripts are managed in this section. Devices that are used by the script must be declared in the first line of the code. Once declared, the devices can be used as if imported and can be used as objects by their specified name. if the name is unclear, check the "Devices Overview"! This process works by injecting the relevant import statements of the used device clients into the script.

Whenever a script is executed, it is accompanied by an initialization script and a termination script. These scripts can be written by an admin user when adding a new device or can be changed at a later stage in the script list in this view. These scripts are not obligatory and experimental scripts can be run even without them. However, it is recommended to implement them to ensure defined starting conditions when addressing a device. Some settings may conflict with what is stated in the user script and may lead to script failure. In case of a script failure and at the end of an experiment, the termination script will execute the commands to reset the device to its default behaviour. In case of delicate equipment, this can prevent damage.

Data Handler

Whenever laboratory equipment is run, data is generated. Data integrity ("Fair data") is becoming increasingly important in the industrial as well as the academic work environment. The device manager offers database integration that will make it easy to harness all information produced during an experiment. Database configuration and individual settings for eachd evice can be done in the Data Handler view. We differentiate between data and meta-data. Meta data is as important as normal process data. It is necessary to put the recorded data in perspective, especially for people who didn't perform the experiment themselves. Hence, meta-data should be captured. The problem that arises is: How do we tell the difference between data and meta-data? In the device detail view of the data-handler, the user can set checkboxes to do this. Furthermore, all SiLA_Service functions are automatically marked as 'meta-data'.

Polling intervals of data and meta-data can be specified in the device detail view of the Data Handler tab if the default values of 30s and 1h respectively do not fit the task at hand. A configuration file with the specified polling intervals of each function of a device will be generated in the local tmp folder. There is the option to export and import these files to make life easier. In case specific experiments require a specific interval configuration, a configuration file can be supplied.

Log

The Log view displays the logging information of the backend server. All classes of logging statements (INFO, WARNING, CRITICAL, ERROR) are shown. A selection menu helps the user to narrow down the shown logging statements by specifying the time and kind of the statement.
